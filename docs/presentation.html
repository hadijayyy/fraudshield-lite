Limitations of Current Approaches

  • Rules-based engines — static thresholds, easily bypassed by adaptive fraudsters
  • High false positives — 89% of flagged transactions are legitimate, degrading UX
  • No explainability — compliance (AML) requires justification for each decision
  • No time-awareness — models trained on shuffled data leak future information
  • Class imbalance ignored — fraud is ~0.1% of transactions; naive models predict all as legitimate

Feature Engineering

36 hand-crafted features across four categories capturing transaction, temporal, and behavioral patterns

💰 Balance Features (12)
  • balance_diff_orig — change in origin balance before/after
  • balance_diff_dest — change in destination balance
  • orig_balance_drained — binary: origin near-zero after tx
  • amount_to_orig_ratio — transaction amount / origin balance
  • amount_to_dest_ratio — transaction amount / dest balance
  • orig_balance_ratio, dest_balance_ratio
⚡ Velocity Features (14)
  • orig_tx_count_24h — origin tx count in past 24h
  • orig_tx_count_7d — origin tx count in past 7 days
  • orig_sum_amount_24h, orig_sum_amount_7d
  • orig_avg_amount_24h, orig_avg_amount_7d
  • dest_tx_count_24h, dest_tx_count_7d
  • dest_sum_amount_24h, dest_sum_amount_7d
⏰ Temporal Features (6)
  • hour_of_day — cyclic encoded (sin + cos)
  • day_of_week — cyclic encoded (sin + cos)
  • is_weekend — binary flag
  • hour_sin, hour_cos, dow_sin, dow_cos
🏷️ Type Encoding (4)
  • One-hot encoding of type (5 transaction types)
  • CASH_IN, CASH_OUT, DEBIT, PAYMENT, TRANSFER
  • Frequency encoding: type_freq
  • Target encoding: type_target_enc

Why XGBoost?

  • Handles missing values natively — no imputation needed for velocity features on first transactions
  • Built-in regularization — prevents overfitting on rare fraud patterns
  • Native support for scale_pos_weight for cost-sensitive training (set to sum(neg)/sum(pos) ≈ 910)
  • Tree-based models capture non-linear interactions (e.g., amount_to_orig_ratio × balance_diff_orig)

Key Design Decisions

  • FastAPI — async I/O, automatic OpenAPI docs, Pydantic validation for request/response schemas
  • XGBoost JSON serialization — portable model artifact, no pickling issues across Python versions
  • SHAP TreeExplainer — exact Shapley values for tree models, O(1) per prediction via path-dependent algorithm
  • Streamlit — lightweight dashboard for fraud analysts with real-time alerting and manual review queue
  • Docker multi-stage build — separate build and runtime images, final image <250MB

SHAP Explainability

Per-transaction feature attribution enabling AML-compliant decision justification

Top 10 Features by SHAP Importance

Rank Feature Category Mean |SHAP|
1 balance_diff_orig Balance 0.342
2 amount_to_orig_ratio Balance 0.287
3 orig_balance_drained Balance 0.194
4 amount Raw 0.156
5 orig_tx_count_24h Velocity 0.098
6 orig_sum_amount_24h Velocity 0.071
7 balance_diff_dest Balance 0.058
8 is_cash_out Type 0.042
9 dest_tx_count_24h Velocity 0.031
10 hour_sin Temporal 0.019

Interpretation

  • balance_diff_orig — fraudsters drain accounts completely; a large drop in origin balance is the #1 signal
  • amount_to_orig_ratio — fraudulent transactions often transfer nearly 100% of available balance
  • orig_balance_drained — accounts left with near-zero balance post-transaction are highly suspicious
  • Velocity features — abnormally high 24h transaction counts and total amounts indicate mule behavior
  • CASH-OUT type — fraudsters prefer immediate liquidation; cash-out transactions are 2× more likely to be fraud

AML Compliance: SHAP provides per-transaction feature contributions, satisfying regulatory requirements for explainable decisions (EU AI Act, OJK regulations). Each fraud alert includes a ranked list of contributing factors.
